## 7. 기업 데이터 거버넌스 4대 기둥 + 영구적 Ingestion

| 기둥 | 핵심 |
|------|------|
| Security | 기밀 데이터의 무단 ingestion 차단 |
| Privacy | GDPR, EU AI Act 등 규제 준수 |
| Bias | 데이터 수집·학습 전 과정의 편향 관리 |
| Ethics | 조직 문화와 의사결정 체계의 윤리성 |

**영구적 Ingestion의 위험**: 한 번 ingestion된 데이터는 모델 내부에 영구 흡수되어 삭제 불가. 의도적 extraction 공격이나 일상적 실험에서 기밀 노출 가능.

**RAG**: 모델 내부에 데이터를 영구 ingestion하지 않고, 외부 API로 실시간 조회하여 contextual grounding을 제공하는 아키텍처. 폐쇄적 환경과 공개 지식 접근 사이의 딜레마를 해결.

## 10. Trust Boundary 5대 경계

| # | 경계 | 핵심 위험 |
|---|------|----------|
| 1 | Model (Public API vs 로컬) | 민감 데이터 노출 / Supply Chain Risk |
| 2 | User Interaction (입출력) | Prompt Injection / 독성 데이터 생성 |
| 3 | Training Data (내부 vs 외부) | PII 노출 / Data Poisoning |
| 4 | Live External Data (실시간) | 간접 Prompt Injection |
| 5 | Internal Services (DB/백엔드) | 권한 상승 / 대규모 유출 |

**결론**: 보안의 핵심 = 모델 자체가 아닌 **Trust Boundary의 식별과 통제**

## 11. Prompt Injection

**정의**: 공격자가 악의적 입력을 통해 LLM의 자연어 이해 능력을 조작하여, 의도된 가이드라인을 벗어나게 만드는 해킹 기법 (OWASP LLM Top 10 1위)

**LLM의 본질적 취약성 3가지**:
1. 명령어(Instruction)와 데이터(Data)를 동일한 채널로 처리
2. 뛰어난 맥락 이해력이 오히려 공격자의 심리전에 취약점으로 작용
3. 유동적 출력(Fluid Nature) 특성으로 모든 엣지 케이스 테스트 불가

**Direct Injection**: 사용자가 직접 입력을 조작 → Jailbreak 유도. 4대 기법: Forceful Suggestion / Reverse Psychology / Misdirection / Automated Adversarial Prompting. 각 기법의 원리와 대표 사례를 설명할 수 있어야 한다.

**Indirect Injection**: 외부 데이터 소스(웹, 문서)에 은닉된 악성 명령을 LLM이 처리하면서 발생. 사용자의 직접적 악의 없이도 시스템 침해가 가능하다는 점이 핵심. 이 현상을 설명하는 보안 용어와 그 의미를 알아야 한다.

## 12. Data Poisoning

**정의**: Training Data를 오염시켜 모델에 취약점·편향·비윤리적 행동을 주입

**핵심 사례**: Microsoft Tay (2016) — 4chan의 조직적 캠페인으로 24시간 만에 혐오 발언 봇으로 변질. "Repeat after me" 취약점 + 대량 독성 데이터 주입(Flooding with rhetoric)

**교훈**: 2016~2024, 7년간 본질적 취약점 미해결. 채택 속도만 기하급수적 증가.

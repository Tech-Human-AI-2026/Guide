# 인간중심의AI 기말고사 복습 해설

이 문서는 `기말고사_복습자료.md`를 읽을 때 놓치기 쉬운 연결 관계와 오해를 풀어 쓴 해설이다. 시험 범위는 7주차 평가 하네스, 9주차 토큰 경제학, 10주차 구조화 출력과 가드레일이다.

---

## 1. 평가 하네스와 AI Safety

> "안전하다"는 말은 평가 방법이 없으면 주장일 뿐이다. 평가 하네스는 그 주장을 반증 가능한 절차로 바꾸는 장치다.

**D, T, M, G를 한 문장으로 외우는 법**: 데이터셋 D에서 문제를 꺼내고, 템플릿 T로 모델 입력을 만들고, 모델 M이 답을 생성하고, 채점기 G가 정답과 비교한다. 시험에서는 네 요소의 이름만 묻기보다 "무엇이 바뀌면 점수가 흔들리는가"를 물을 수 있다. 예를 들어 같은 TruthfulQA라도 `Q: ... A:` 템플릿을 쓰는지, few-shot 예시를 붙이는지, LLM judge를 쓰는지에 따라 점수가 달라진다.

**lm-eval-harness**: 학술 벤치마크 표준 구현에 가깝다. "task YAML 하나에 데이터 로딩, prompt, metric을 적는다"는 이미지를 기억하면 된다. Hugging Face Open LLM Leaderboard 같은 비교 작업에 자연스럽다.

**HELM**: 단일 점수의 함정을 피하려는 도구다. 모델 A가 accuracy는 높지만 toxicity가 높거나 robustness가 낮을 수 있다. 인간중심 AI에서는 정확도 하나로 모델을 평가하면 안 되므로, HELM의 다축 평가 사고가 중요하다.

**Inspect AI**: tool use, multi-turn, model-graded eval처럼 에이전트형 평가를 짜기 좋은 범용 프레임워크다. AISI가 사용하는 위험 평가 카테고리와 Inspect 자체의 구조를 혼동하지 말아야 한다. bio, cyber, autonomy는 Inspect로 평가할 수 있는 영역이지 Inspect의 내부 구성요소 이름이 아니다.

**Goodhart's Law**: 벤치마크가 목표가 되면 모델은 실제 능력보다 점수 올리는 방식에 최적화될 수 있다. "점수가 높다"와 "현실에서 안전하다"는 같은 말이 아니다.

**Benchmark contamination**: 평가 문제가 학습 데이터에 들어가 있으면 모델은 추론한 것이 아니라 기억했을 수 있다. held-out 평가셋과 canary string은 이 문제를 줄이려는 방법이지만 완전한 해결책은 아니다.

**LLM-as-judge 편향**: 채점기가 LLM이면 position bias, verbosity bias, self-enhancement bias가 생길 수 있다. 길고 그럴듯한 답이 더 높은 점수를 받거나, judge 모델과 같은 계열의 답이 유리할 수 있다.

**RSP와 임계값 정책**: Responsible Scaling Policy의 핵심은 "능력이 높으니 막연히 위험하다"가 아니라 "사전에 정한 능력 임계값을 넘으면 보호조치 없이는 진행하지 않는다"는 조건부 약속이다. 따라서 ASL-3 임계값을 넘었는데 ASL-3 보호 조치가 없으면 배포를 제한하는 판단이 자연스럽다.

**Constitutional AI**: 모델이 헌법 원칙을 기준으로 자기 답변을 비판하고 수정한다. 평가가 학습 루프 안으로 들어간 사례지만, judge 역할을 하는 모델이 틀릴 수 있다는 한계도 같이 기억해야 한다.

## 2. 토큰 경제학

> 토큰 비용은 단순 청구서 문제가 아니다. 운영 비용이 높으면 좋은 AI 도구를 쓸 수 있는 사람과 조직이 제한되므로 포용성 문제가 된다.

**입력과 출력 비용의 비대칭**: 출력 토큰은 보통 입력 토큰보다 비싸다. 입력은 병렬 처리할 수 있지만 출력은 다음 토큰을 하나씩 생성하기 때문이다. 긴 답변을 요구하는 prompt는 품질 문제가 아니라 비용 공격 표면이 될 수 있다.

**Prompt caching의 핵심**: 같은 prefix가 반복될 때 prefix 토큰의 K와 V를 재사용한다. Q가 아니라 K,V다. 새 토큰은 자기 Q로 과거 토큰들의 K를 조회하고 V를 섞는다. 따라서 긴 시스템 프롬프트, 도구 정의, rubric처럼 반복되는 prefix가 길수록 caching 효과가 커진다.

**TTL 해석**: TTL은 cache entry가 살아 있는 시간이다. hit이 발생하면 연장될 수 있다. 10:00에 cache write가 생기고 10:03, 10:07에 hit이 나면 계속 살아 있지만, 10:07 이후 13분이 지난 10:20 요청은 5분 TTL 기준으로 miss가 된다.

**비용 손계산 포인트**: caching은 출력 비용을 줄이지 않는다. 예제에서 전체 비용이 1.08달러에서 0.5469달러로 줄어드는 이유는 입력 prefix 비용이 줄었기 때문이다. 출력 300토큰을 100번 생성하는 비용 0.45달러는 그대로 남는다.

**좋은 prefix 설계**: 도구 정의, 시스템 정책, 긴 공통 문서를 앞쪽에 두고 사용자 질문, 사용자 ID, 현재 시각, 요청 ID는 뒤쪽에 둔다. 현재 시각을 prefix 앞에 넣으면 매 요청마다 prefix hash가 바뀌어 cache hit이 깨진다.

**LiteLLM의 핵심**: LiteLLM은 싼 모델을 고르는 도구로만 보면 부족하다. 비용, privacy, latency, 기능 요구, fallback 정책을 실행하는 gateway다. 민감 정보가 포함된 요청은 local 또는 승인된 provider로만 보내야 하고, fallback도 같은 보안 등급 안에서만 허용해야 한다.

**잘못된 fallback의 위험**: local 모델이 timeout됐다고 성적표를 cheap cloud 모델로 보내면 서비스는 살아나지만 데이터 경계가 무너진다. 이 상황은 Sensitive Information Disclosure와 Supply Chain 위험이 동시에 생긴다.

**Helicone의 가치와 위험**: Helicone은 비용 대시보드, prompt versioning, 이상 탐지를 제공한다. 하지만 prompt와 response를 저장하면 로그 자체가 민감 데이터 저장소가 된다. observability는 방어 도구이면서 관리해야 할 데이터 자산이다.

**OWASP 연결**: 긴 출력과 반복 호출은 Model DoS 또는 Unbounded Consumption이다. LLM output을 브라우저, DB, shell, tool에 검증 없이 넘기는 것은 Improper Output Handling이다. cache, log, prompt에 민감 정보가 들어가는 문제는 Sensitive Information Disclosure다.

## 3. 구조화 출력과 가드레일

> 구조화 출력은 환각을 없애는 마법이 아니다. LLM 답변이 시스템 행동으로 넘어가기 전에 검증 가능한 계약을 통과하게 만드는 장치다.

**구조화 출력의 목적**: 사람에게 자연스러운 문장보다 시스템이 안전하게 처리할 수 있는 데이터가 필요할 때 쓴다. 환불 챗봇이라면 `eligible`, `amount_krw`, `evidence_ids`, `risk_level`, `needs_human_review` 같은 필드를 강제한다.

**Instructor가 잘하는 것**: Pydantic 모델을 이용해 필수 필드, 타입, enum, 범위, 최소 길이를 검증한다. JSON 파싱 실패와 누락 필드를 줄이고, 실패하면 오류 맥락을 넣어 재시도할 수 있다.

**Instructor가 못하는 것**: `evidence_ids`가 실제 승인 문서인지, `reason`이 법적으로 맞는지, 환불 정책 해석이 사실인지까지 보장하지 않는다. 타입이 맞는 거짓말도 가능하다. 그래서 RAG 근거 검증, 문서 버전 확인, 사람 검토, 감사 로그가 필요하다.

**JSON mode와 schema validation의 차이**: JSON mode는 `{}`와 따옴표 같은 문법을 맞추는 데 도움이 된다. 그러나 `amount_krw`가 `"8만원쯤"`이면 JSON 문법은 맞아도 integer schema에는 실패한다. schema validation은 문법 이후의 계약 검사다.

**Guardrails AI**: 입력이나 출력에 validator를 붙이는 데 강하다. validator는 정규표현식만 뜻하지 않는다. PII 탐지, 독성 분류, 길이 제한, JSON 유효성, custom Python 함수, LLM 기반 판정 모두 validator가 될 수 있다.

**실패 처리 선택**: 형식 오류는 reask가 적합하고, PII는 fix 또는 mask가 적합할 수 있다. 그러나 정책 판단 오류를 자동으로 fix하면 위험하다. "아마 환불 가능"을 `eligible=true`로 바꾸는 것은 단순 포맷 수정이 아니라 업무 판단 변경이다.

**NeMo Guardrails**: 대화 흐름과 rail 위치를 설계하는 데 강하다. 환불 질문인지 intent를 잡고, 주문번호가 없으면 묻고, RAG chunk를 거르고, action 호출 전 권한과 금액을 검사하고, 출력 전 PII를 다시 검사하는 전체 흐름을 다룬다.

**Guardrails AI와 NeMo의 차이**: Guardrails AI는 "이 입력이나 출력이 검사를 통과하는가"에 강하고, NeMo는 "대화가 어떤 flow를 따라가야 하며 어느 지점에서 어떤 rail을 실행할 것인가"에 강하다. 복잡한 상담 챗봇은 NeMo로 flow를 잡고 Guardrails AI validator와 Pydantic 검증을 조합하는 구성이 자연스럽다.

**세 겹 방어의 핵심**: 입력 rail, 검색/RAG, LLM 생성, 구조화 출력 검증, 출력 rail, 애플리케이션 권한 계층을 순서대로 둔다. 중요한 것은 LLM이 항상 맞는다고 믿는 것이 아니라, 틀렸을 때도 DB 삭제, 환불 실행, 의료 조언, 법적 약속으로 바로 이어지지 않게 막는 것이다.

**Misinformation 대응**: schema는 환각의 형식을 정리할 수 있지만 사실성을 보장하지 않는다. 고위험 도메인에서는 RAG, 근거 문서 ID 검증, 사람 검토가 필요하다.

## 4. 헷갈리기 쉬운 구분

| 헷갈리는 쌍 | 구분 |
|---|---|
| 평가 하네스 vs 가드레일 | 하네스는 모델을 평가하는 시험대, 가드레일은 운영 중 입력·검색·도구·출력을 통제하는 방어 계층 |
| HELM vs Helicone | HELM은 모델 평가 방법론, Helicone은 운영 호출 observability |
| Inspect AI vs LiteLLM | Inspect AI는 평가 프레임워크, LiteLLM은 provider 라우팅 gateway |
| Prompt caching vs 출력 재사용 | caching은 입력 prefix K,V 재사용, 이미 생성된 답변 재사용이 아님 |
| JSON mode vs 구조화 출력 검증 | JSON mode는 문법 보조, 구조화 검증은 필드·타입·enum·범위 검사 |
| Instructor vs NeMo | Instructor는 호출 1개의 출력 계약, NeMo는 대화 flow와 rail 오케스트레이션 |
| Observability vs 보안 보장 | observability는 탐지와 분석을 돕지만, 로그 보안 정책 없이는 새 위험이 됨 |

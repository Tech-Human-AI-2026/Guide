## 1. 요약

학생들의 공통 결론은 분명하다. `lm-evaluation-harness`는 LLM 평가 실행을 표준화하는 강력한 도구이지만, 이것만으로 평가가 자동으로 공정해지거나 실제 서비스 성능을 보장하지는 않는다. 표준 벤치마크, YAML 기반 태스크 정의, 다양한 모델 백엔드, 결과 재현성이라는 장점은 크지만, 프롬프트 민감도, 정적 벤치마크 포화, 데이터 오염, 자유 생성 채점의 어려움, 분산 실행과 비용 문제는 계속 남는다.

특히 많이 반복된 문제의식은 세 가지다.

1. **평가 실행의 표준화와 평가 타당성은 다르다.** 같은 프레임워크를 써도 prompt, few-shot 예시, chat template, backend, post-processing이 다르면 점수 비교가 흔들린다.

3. **다음 단계는 평가 기록과 평가 방식의 고도화다.** 실행 manifest, 의미론적 YAML 검증, LLM-as-a-Judge, 동적 벤치마크, 데이터 오염 탐지, 에이전트 평가 환경이 핵심 개선축으로 제안됐다.

## 4. 학생들이 찾은 핵심 한계점

### 4.1 프롬프트와 실행 환경에 민감한 재현성

대부분의 학생이 가장 먼저 지적한 문제는 재현성이다. `lm-evaluation-harness`는 같은 task를 같은 방식으로 실행하게 해주지만, 실제 점수는 prompt template, few-shot 예시 순서, chat template, stop sequence, decoding parameter, tokenizer, backend 구현, 라이브러리 버전, GPU 환경에 영향을 받는다.

따라서 "같은 task name을 실행했다"는 사실만으로는 충분하지 않다. 비교 가능한 평가는 최소한 모델 버전, 데이터셋 버전, prompt, few-shot 샘플, generation args, backend, post-processing, seed, hardware, CUDA/driver 버전까지 함께 기록해야 한다.

### 4.2 정적 벤치마크의 포화와 실제 활용 성능과의 괴리

학생들은 MMLU, GSM8K, ARC 같은 표준 벤치마크가 논문 비교에는 유용하지만, 실제 서비스 품질을 충분히 설명하지 못한다고 보았다. 최신 모델이 기존 시험지에서 높은 점수를 얻더라도, 실제 대화의 맥락 유지, RAG의 근거 활용, 도구 호출, 장기 계획, 실패 복구 능력은 별도로 평가해야 한다.

이 지점에서 "정적 데이터셋 기반 모델 평가"와 "동적 환경 속 AI 시스템 평가"를 구분해야 한다는 의견이 강했다.

### 4.3 자유 생성 답변 채점의 어려움

객관식이나 loglikelihood 기반 평가는 자동화하기 쉽다. 그러나 요약, 코드 리뷰, 설득력 있는 글쓰기, 복잡한 추론 설명처럼 정답이 하나로 고정되지 않는 태스크에서는 Exact Match나 단순 문자열 매칭이 부적합하다.

LLM-as-a-Judge가 대안으로 많이 제안됐지만, 학생들은 이 방식도 완전한 해결책은 아니라고 보았다. judge 모델은 위치 편향, 장황성 편향, 자기 선호 편향, 프롬프트 편향을 가질 수 있다. 따라서 judge 모델 이름, 버전, 평가 프롬프트, temperature를 저장하고, 가능하면 복수 judge와 인간 표본 평가로 보정해야 한다.

### 4.4 데이터 오염과 굿하트의 법칙

공개 벤치마크는 모델 학습 데이터에 포함될 수 있다. 이 경우 높은 점수가 실제 일반화 능력인지, 이미 본 문제를 맞힌 것인지 구분하기 어렵다. 또 리더보드 점수가 목표가 되면 모델 개발이 실제 문제 해결보다 특정 벤치마크 최적화로 흐를 위험이 있다.

학생들은 데이터 오염 탐지, 동적 문제 생성, 비공개 테스트셋, 주기적으로 갱신되는 벤치마크가 필요하다고 보았다.

### 4.5 인프라 비용과 분산 평가 한계

대형 모델 평가에는 GPU 메모리, 실행 시간, 저장 공간, API 비용이 많이 든다. 학생들은 특히 멀티노드 분산 평가, vLLM/DeepSpeed/Ray 같은 실행 환경 연계, Apple Silicon MPS 환경 안정성, 평가 중 진행률과 GPU 메모리 모니터링 부족을 문제로 봤다.

이 한계는 단순 편의성 문제가 아니다. 실행 환경이 다르면 점수 자체가 조금씩 달라질 수 있으므로, 인프라 정보는 평가 결과의 일부로 기록되어야 한다.

### 4.6 YAML 기반 확장성의 양면성

YAML은 새 task를 빠르게 정의할 수 있게 해주는 장점이다. 하지만 학생들은 "문법적으로 올바른 YAML"과 "평가 의미상 올바른 YAML"은 다르다고 지적했다. 예를 들어 metric과 output type이 맞지 않거나, split이 없거나, prompt가 모호하거나, 후처리 규칙이 답을 왜곡할 수 있다.

고급 평가를 붙이려면 Python 코드, sandbox, 외부 API, 커스텀 metric이 필요해지는데, 이때 프레임워크 내부를 깊게 이해해야 하는 진입장벽도 반복적으로 언급됐다.

### 4.7 멀티모달과 한국어 평가의 부족

일부 학생은 멀티모달 지원이 아직 중심 기능이 아니라고 지적했다. 이미지, 오디오, 비디오가 포함된 LMM 평가를 다루려면 별도 프로젝트나 확장 계층이 필요하다.

반면 한국어·비영어권 평가 부족은 한 학생만 명시적으로 다뤘다. 실제 Human-AI 수업 관점에서는 이 부분을 더 키울 필요가 있다. 국내 서비스나 공공 영역에서 쓸 모델을 평가하려면 한국어 지식, 존댓말, 행정·법률·의료 등 로컬 도메인 표현, 문화적 맥락을 별도 평가해야 한다.

## 5. 학생들이 제안한 개선 방향

### 5.1 Evaluation Manifest 강제

가장 실용적인 개선안은 평가 실행 manifest를 강제하는 것이다. 점수만 저장하지 말고 다음 정보를 한 묶음으로 남겨야 한다.

```yaml
evaluation_manifest:
  model_name: ...
  model_revision: ...
  lm_eval_version: ...
  task_name: ...
  dataset_name: ...
  dataset_revision: ...
  prompt_template_hash: ...
  fewshot_examples_hash: ...
  generation_args: ...
  backend: ...
  tokenizer: ...
  post_processing: ...
  random_seed: ...
  hardware:
    gpu: ...
    driver: ...
    cuda: ...
    vram: ...
  result_files:
    samples: ...
    metrics: ...
```

이 manifest가 있어야 논문, 리더보드, 수업 과제에서 "같은 조건으로 비교했다"는 말을 검증할 수 있다.

### 5.2 YAML 의미론적 검증과 task lint

학생들이 반복해서 제안한 두 번째 개선은 YAML 검증의 고도화다. 단순 문법 검사에 그치지 않고 다음을 검사해야 한다.

- `output_type`과 `metric` 조합이 맞는가
- dataset split이 실제 존재하는가
- few-shot 개수가 데이터셋 조건과 충돌하지 않는가
- prompt가 선택지를 누락하거나 정답 힌트를 포함하지 않는가
- 후처리 regex가 특정 형식 답변만 과도하게 선호하지 않는가
- 필요한 hardware, memory, API key, sandbox 조건이 명시됐는가

이는 평가 실패를 줄이는 기능이면서, 수업 과제에서는 학생들이 좋은 평가 task를 설계하도록 돕는 교육 도구가 될 수 있다.

### 5.3 프롬프트 강건성 평가

하나의 prompt에서 얻은 점수만 제출하는 대신, 여러 prompt 변형에서의 평균, 표준편차, 최악 점수, 신뢰구간을 함께 제시해야 한다. 예를 들어 공백, 문장 순서, few-shot 예시 순서, 선택지 표기, 한국어/영어 표현을 바꿔도 점수가 안정적인지 확인하는 방식이다.

이 접근은 "모델이 문제를 푼 것인지, 특정 prompt 형식에 맞춘 것인지"를 구분하는 데 도움이 된다.

### 5.4 LLM-as-a-Judge의 공식 메트릭화

학생들은 자유 생성 답변을 평가하기 위해 `llm_judge` 같은 metric type을 공식화하자고 제안했다. 단, judge를 쓸 때는 다음 절차가 필요하다.

- 평가 rubric을 명시한다.
- judge 모델명, 버전, temperature, prompt를 저장한다.
- 답변 순서를 바꿔 위치 편향을 점검한다.
- 복수 judge 또는 인간 표본 평가와 비교한다.
- judge 비용과 지연시간을 기록한다.

LLM-as-a-Judge는 정답이 하나로 고정되지 않는 태스크에 필요하지만, 편향 통제 없이는 또 다른 불투명한 평가가 될 수 있다.

### 5.5 동적 벤치마크와 오염 탐지

고정 벤치마크만 쓰지 말고 평가 시점에 문제를 갱신하거나 합성하는 방식이 제안됐다. 예를 들어 최신 뉴스, 비공개 문서, 수업 중 생성한 데이터, 내부 정책 문서를 바탕으로 매회 다른 문제를 만들 수 있다.

동시에 n-gram overlap, perplexity anomaly, 공개 데이터셋 fingerprint 등을 이용해 모델이 평가 데이터를 이미 봤을 가능성을 경고하는 오염 탐지 단계가 필요하다.

### 5.6 RAG·에이전트·멀티턴 평가 환경

학생들은 차세대 평가가 단순 QA에서 벗어나야 한다고 보았다. 평가 대상이 모델 하나가 아니라 시스템이라면 다음 요소를 포함해야 한다.

- RAG: 검색 문서 적절성, 답변의 근거 충실도, hallucination, citation 품질
- 멀티턴: 이전 대화 기억, 맥락 유지, 지시 변경 대응
- 에이전트: 관측, 계획, 도구 호출, 실패 복구, 최종 성공률
- 코드 실행: sandbox, timeout, Pass@k, 보안 제약

이런 평가는 YAML만으로 끝나기 어렵기 때문에 Python plugin, sandbox, 외부 simulator와 연결되는 구조가 필요하다.

### 5.7 인프라 관측성과 분산 실행 개선

멀티노드 평가, vLLM/Ray/DeepSpeed 연동, GPU 메모리 모니터링, ETA, 실패 재개, cache 계층 분리가 제안됐다. 특히 대형 모델 평가에서는 "실행 가능성" 자체가 평가 설계의 일부다.

수업 과제에서도 결과 점수뿐 아니라 실행 환경, 실패 로그, 비용 추정, 실행 시간을 함께 제출하게 하면 평가 인프라를 더 현실적으로 이해할 수 있다.

### 5.8 한국어·비영어권 평가 확장

이번 제출에서 상대적으로 약했던 부분이다. 향후에는 한국어 평가를 별도 축으로 다루는 것이 좋다. 예를 들어 Ko-MMLU 같은 공개 벤치마크뿐 아니라, 한국 행정문서, 학교 공지, 연구계획서, 상담 대화, 존댓말/반말 전환, 지역 문화 지식 등을 포함한 과제를 설계할 수 있다.

## 7. 전체 수업 피드백

이번 제출의 좋은 점은 학생 대부분이 단순히 "좋은 평가 도구"라고 끝내지 않고, 평가의 한계와 사회적·실무적 위험까지 연결했다는 점이다. 특히 정적 벤치마크의 한계, 프롬프트 민감도, 데이터 오염, LLM-as-a-Judge의 필요성은 수업 전체가 공통 언어로 가져갈 만하다.

아쉬운 점도 분명하다.

- 실제 `lm_eval` 실행 결과, 로그, config, output sample을 제출한 사례는 거의 보이지 않았다.
- URL 출처가 있는 repo가 4/9개뿐이라, 일부 주장은 근거 추적이 어렵다.
- 한국어·비영어권 평가, 접근성, 공정성, 사용자 피해 관점은 아직 충분히 확장되지 않았다.
- 멀티모달과 에이전트 평가는 많이 언급됐지만, 구체적인 task 설계 예시는 부족했다.

다음 과제로 이어간다면 "분석 보고서"에서 한 단계 더 나아가, 작은 모델 하나와 작은 task 하나를 실제로 `lm-evaluation-harness`로 실행하고 manifest, prompt, output sample, 실패 로그까지 제출하게 하는 것이 좋다. 그 다음 같은 task를 prompt 변형 3개로 돌려 표준편차를 비교하게 하면, 이번 주에 말한 프롬프트 민감도와 재현성 문제가 실제로 체감될 것이다.

## 8. 공유용 결론

`lm-evaluation-harness`는 LLM 평가 생태계에서 중요한 표준 도구다. 하지만 좋은 평가 도구를 쓴다고 좋은 평가가 자동으로 만들어지지는 않는다. 앞으로의 핵심은 점수 하나를 더 얻는 것이 아니라, 평가 조건을 투명하게 기록하고, 정적 벤치마크를 동적·상호작용 평가로 확장하며, 자유 생성 답변과 실제 서비스 맥락을 더 정교하게 평가하는 것이다.

학생들이 제안한 개선 방향을 하나로 요약하면 다음과 같다.

> 평가를 실행하는 도구에서, 평가 조건·편향·환경·실제 사용 맥락까지 함께 관리하는 평가 인프라로 발전해야 한다.

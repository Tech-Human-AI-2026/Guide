## 1. 평가 하네스와 AI Safety

### 1-1. 평가 하네스의 의미

`"이 모델이 안전하다"`는 주장은 평가 방법이 명시되지 않으면 검증 불가능하다. 평가 하네스는 같은 모델, 같은 프롬프트, 같은 채점 규칙으로 다시 돌릴 수 있는 코드화된 시험대다.

| 요소 | 의미 | 예 |
|---|---|---|
| D: 데이터셋 | 입력과 정답 또는 정답 패턴 | MMLU, TruthfulQA, AdvBench |
| T: 프롬프트 템플릿 | 입력을 모델에게 어떤 형식으로 줄지 | `Q: {question}\nA:` |
| M: 모델 호출 규칙 | temperature, top_p, max_tokens, system prompt | `temperature=0` |
| G: 채점기 | 모델 출력을 정답과 비교 | exact match, regex, LLM judge |

평가 점수의 뼈대:

```text
x_i -> T(x_i) -> M(T(x_i)) = y_hat_i -> G(y_hat_i, y_i) -> 평균 s
```

핵심은 D, T, M, G가 코드와 설정으로 고정되어 재현 가능해야 한다는 점이다.

### 1-2. 대표 도구 비교

| 도구 | 핵심 성격 | 기억할 점 |
|---|---|---|
| lm-evaluation-harness | 학술 벤치마크 표준 구현 | task YAML로 데이터, prompt, metric을 정의 |
| HELM | 다축 평가 | accuracy만 보지 않고 calibration, robustness, bias, toxicity, efficiency 등을 함께 봄 |
| Inspect AI | 범용 평가 인프라 | tool use, multi-turn, model-graded eval, trace 기반 평가에 강함 |

### 1-3. RSP와 Constitutional AI

Anthropic Responsible Scaling Policy(RSP)는 모델 능력을 ASL로 분류하고, 사전에 정한 임계값을 넘으면 보호 조치 없이는 학습 또는 배포를 제한하는 조건부 약속이다. 중요한 구조는 `능력 평가 -> 임계값 비교 -> 보호조치 발동`이다.

Constitutional AI는 사람이 모든 안전 라벨을 달지 않고, 모델이 헌법 원칙에 비추어 자기 답변을 비판하고 수정하게 하는 학습 방식이다. 이때 모델은 LLM-as-judge 형태의 채점기 역할을 하므로, 평가가 학습 루프 안으로 들어간 사례다.

### 1-4. 평가의 세 가지 한계

| 한계 | 의미 | 시험 키워드 |
|---|---|---|
| Goodhart's Law | 벤치마크가 목표가 되면 실제 능력보다 점수 최적화가 앞설 수 있음 | benchmark gaming, sandbagging |
| Benchmark contamination | 평가 데이터가 학습 데이터에 섞이면 푼 것이 아니라 기억한 것일 수 있음 | held-out eval, canary string |
| Eval gaming | LLM judge가 편향을 가질 수 있음 | position bias, verbosity bias, self-enhancement bias |

## 2. 토큰 경제학

### 2-1. 비용은 거버넌스 문제다

LLM 운영 비용이 높으면 좋은 AI 서비스를 쓸 수 있는 조직과 그렇지 못한 조직 사이의 격차가 커진다. 따라서 prompt caching, routing, budget, observability는 단순 최적화가 아니라 포용성을 지키는 운영 통제다.

입력 토큰과 출력 토큰의 단가는 다르며, 출력 토큰이 더 비싼 경우가 많다. 출력은 다음 토큰을 순차적으로 생성해야 하므로 긴 답변을 유도하는 prompt는 곧 비용 공격 표면이 된다.

### 2-2. Prompt Caching

Transformer attention에서 각 토큰은 Q, K, V를 만든다. Prompt caching의 핵심은 같은 prefix가 반복될 때 prefix 토큰의 K와 V를 재사용하는 것이다.

| 항목 | 의미 |
|---|---|
| cache write | 처음 prefix를 계산해 캐시에 쓰는 것. 일반 입력보다 비쌀 수 있음 |
| cache read | 저장된 prefix를 읽는 것. 일반 입력보다 싸다 |
| TTL | cache entry가 유지되는 유효기간. hit이 발생하면 연장될 수 있음 |
| cache miss | prefix가 없거나 만료되거나 정확히 일치하지 않아 다시 계산하는 상황 |

설계 규칙:

1. 변하지 않는 도구 정의, 시스템 정책, 긴 공통 문서를 앞쪽에 둔다.
2. 사용자 질문, 현재 시각, 사용자 ID, 요청 ID는 뒤쪽에 둔다.
3. 캐시는 비슷한 prompt를 찾아주는 기능이 아니다. breakpoint 앞 prefix가 정확히 같아야 한다.
4. Prompt caching은 입력 prefix 비용을 줄일 뿐, 출력 토큰 비용을 직접 줄이지 않는다.

비용 예제 핵심:

| 경우 | 비용 |
|---|---:|
| caching 없음 | 1.08달러 |
| caching 있음 | 0.5469달러 |
| 입력 비용만 절감 | 0.63달러 -> 0.0969달러 |

출력 비용 0.45달러는 caching 후에도 그대로다.

### 2-3. LiteLLM

LiteLLM은 여러 LLM provider와 로컬 모델 endpoint를 OpenAI 호환 인터페이스로 묶는 gateway다.

| 기능 | 의미 |
|---|---|
| Unified API | provider별 SDK 차이를 숨김 |
| Routing | 비용, latency, 보안 등급, 기능 요구에 따라 모델 선택 |
| Fallback | 장애 시 다른 provider로 재시도 |
| Cost tracking | 사용자, 팀, 프로젝트별 비용과 budget 관리 |

중요한 정책:

- 민감 정보가 있으면 local 또는 승인된 provider로만 보낸다.
- fallback도 같은 보안 등급 안에서만 허용한다.
- 과제 채점이나 법적 판단은 cheap 모델로 보내지 않는다.
- budget 초과 시 모델을 낮추거나 max_tokens를 줄이거나 요청을 거절한다.

LiteLLM은 "무조건 싼 모델" 도구가 아니라 비용, 품질, 프라이버시, 신뢰성의 정책 선택을 실행하는 gateway다.

### 2-4. Helicone

Helicone은 LLM 호출의 비용, latency, token 수, user/session ID, prompt version 등을 기록하는 observability 도구다.

가치:

- 비용 대시보드: 기능별, 사용자별, 환경별 비용 원인 분석
- 프롬프트 버저닝: prompt 변경이 비용과 품질에 미친 영향 비교
- 이상 탐지: 긴 출력, 반복 호출, 특정 사용자 비용 폭증 탐지

주의:

- prompt와 response 로그에는 개인정보, API key, 내부 문서가 들어갈 수 있다.
- redaction, retention, 접근권한, self-host 또는 승인 region 정책이 필요하다.

### 2-5. 9주차 OWASP 매핑

| 주제 | 연결 위험 |
|---|---|
| cache/log/prompt에 민감 정보 포함 | Sensitive Information Disclosure |
| fallback으로 외부 provider에 민감 정보 전송 | Sensitive Information Disclosure + Supply Chain |
| LLM output을 브라우저, DB, shell, tool에 그대로 전달 | Improper Output Handling |
| 긴 출력, 반복 호출, cache miss 유도 | Model DoS / Unbounded Consumption |
| observability 로그가 새 민감 데이터 저장소가 됨 | Sensitive Information Disclosure |

## 3. 구조화 출력과 가드레일

### 3-1. 출력은 문장이 아니라 계약

구조화된 출력은 LLM에게 예쁘게 답하라고 말하는 것이 아니다. 애플리케이션이 받을 수 있는 데이터의 모양을 스키마로 정하고, 필드, 타입, enum, 범위, 필수 근거를 검증하는 것이다.

환불 판정 스키마 예:

| 필드 | 타입 | 의미 |
|---|---|---|
| `eligible` | boolean | 환불 대상 여부 |
| `amount_krw` | integer | 환불 금액 |
| `reason` | string | 판단 이유 |
| `evidence_ids` | list[string] | 근거 문서 ID |
| `risk_level` | enum | `low`, `medium`, `high` |
| `needs_human_review` | boolean | 사람 검토 필요 여부 |

형식이 맞아도 사실이 맞는 것은 아니다. `evidence_ids`가 실제 승인 문서인지, 정책 해석이 맞는지, 법적 책임이 생기는지는 별도 검증이 필요하다.

### 3-2. Instructor와 주변 도구

| 도구 | 자연스러운 사용 장면 | 한계 |
|---|---|---|
| Instructor | LLM API 호출 하나를 Pydantic 객체로 받고 싶을 때 | 사실성, 근거 문서의 진위, 정책 해석을 보장하지 않음 |
| LangChain structured output | chain, agent, tool 흐름의 최종 출력 또는 tool 인자 형식 | workflow 전체 설계가 필요 |
| LlamaIndex structured output | RAG나 문서 추출 결과를 Pydantic 객체로 뽑을 때 | 검색 문서의 품질과 출처 검증이 별도 필요 |

스키마 강제 층:

- 프롬프트 지시: 가장 약함
- JSON mode: JSON 문법에는 도움, schema 위반은 남을 수 있음
- tool/function calling: 함수 인자처럼 구조화
- provider-native structured output: provider가 JSON Schema 등을 직접 받음
- constrained decoding: 문법상 불가능한 토큰 생성을 줄임
- 사후 검증 + 재시도: Pydantic/validator 오류를 넣어 다시 생성
- 사후 검증 + 차단/수정: reject, mask, fix, human review

### 3-3. Guardrails AI

Guardrails AI는 입력/출력 Guard와 validator 조합에 강하다. validator는 정규표현식만이 아니라 PII 탐지, 독성 분류, 길이 제한, JSON 유효성, custom Python 함수, LLM 기반 판정 등을 포함하는 검사 단위다.

실패 처리 방식:

| 방식 | 적합한 상황 |
|---|---|
| exception/reject | 위험 입력, 정책 위반 |
| reask | 형식 오류, 누락 필드 |
| fix | 단순 포맷 오류, PII 마스킹 |
| noop | 실험 또는 로깅 단계 |
| escalate | 고위험 판단, 근거 부족 |

자동 수정은 형식 오류에는 유용하지만 정책 판단 오류에는 위험하다.

### 3-4. NeMo Guardrails

NeMo Guardrails는 대화 flow와 rail 위치를 오케스트레이션하는 데 강하다.

| Rail | 위치 | 역할 |
|---|---|---|
| Input rail | 사용자 입력 직후 | PII, jailbreak, 허용 주제 검사 |
| Dialog rail | 대화 흐름 결정 | intent 해석, 다음 단계 선택 |
| Retrieval rail | RAG 문서 검색 뒤 | 비공개 chunk 제거, 출처 확인 |
| Execution rail | tool/action 호출 전후 | 인자, 권한, 금액, 상태 변경 검사 |
| Output rail | 최종 응답 전 | PII, 독성, 길이, 정책 위반 검사 |

Guardrails AI가 "이 문자열 또는 출력이 규칙을 만족하는가"에 강하다면, NeMo는 "이 발화는 어떤 intent이고 어떤 flow와 rail 순서를 거쳐야 하는가"에 강하다. 실제 시스템에서는 NeMo + Guardrails AI + Instructor/Pydantic + 애플리케이션 권한 계층을 조합한다.

### 3-5. 세 겹 방어 아키텍처

```text
사용자 입력
  -> 입력 레일
  -> 검색/RAG
  -> LLM 생성
  -> 구조화 출력 검증
  -> 출력 레일
  -> 애플리케이션 권한 계층
  -> 사용자 응답 또는 사람 검토
```

핵심은 모델이 아니라 경계다. LLM은 여전히 틀릴 수 있지만, 틀린 답이 곧바로 DB 삭제, 환불 실행, 의료 조언, 법적 약속으로 이어지지 않게 해야 한다.

### 3-6. 10주차 OWASP 매핑

| 위험 | 대표 통제 |
|---|---|
| LLM09 Misinformation | RAG, 근거 검증, 인간 검토, 위험 커뮤니케이션 |
| LLM05 Improper Output Handling | 스키마 검증, output encoding, parameterized query, sandbox |
| LLM04 Data and Model Poisoning | 데이터 출처 추적, 승인 문서, 버전 관리, red team |

JSON schema는 환각을 없애지 않는다. 환각의 형식을 정돈할 수는 있지만, 사실성과 근거성은 별도 계층이 검증해야 한다.

## 4. 시험 직전 체크리스트

- 평가 하네스의 D, T, M, G를 설명할 수 있는가?
- lm-eval-harness, HELM, Inspect AI의 차이를 구분할 수 있는가?
- Goodhart, contamination, LLM-as-judge 편향을 사례로 설명할 수 있는가?
- RSP의 `임계값 -> 보호조치` 구조를 설명할 수 있는가?
- Prompt caching에서 왜 K,V가 재사용되는지 설명할 수 있는가?
- TTL hit/miss와 cache write/read 비용을 손계산할 수 있는가?
- LiteLLM 라우팅에서 privacy fallback 정책을 판단할 수 있는가?
- Helicone의 장점과 로그 보안 위험을 동시에 말할 수 있는가?
- 구조화 출력이 보장하는 것과 보장하지 않는 것을 구분할 수 있는가?
- Guardrails AI와 NeMo Guardrails의 역할 차이를 설명할 수 있는가?
- 입력 rail, RAG, LLM 생성, 구조화 검증, 출력 rail, 권한 계층의 순서를 말할 수 있는가?
